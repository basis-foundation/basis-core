# v0.1 Readiness Review

This document is a point-in-time assessment of basis-core's readiness for a stable v0.1 release. It answers the canonical readiness questions, documents open questions and their dispositions, and records what work was intentionally deferred.

**Review date**: 2026-05-27
**Version assessed**: 0.1.0
**Test suite**: 625 tests passing (pytest, ruff clean, mypy strict, no issues)

---

## What is basis-core?

basis-core is the **authorization kernel** for the BASIS ecosystem — an isolated Python library that evaluates authorization requests and records the results as structured audit events.

It answers a single question: *May subject S perform action A on resource R?* It does nothing else.

The kernel contains: canonical domain types (`Subject`, `Resource`, `Action`, `IdentityContext`), the policy evaluation engine, the enforcement point, audit event contracts, the audit writer protocol, and adapter lifecycle contracts. It depends on nothing except Python 3.10+ and Pydantic ≥ 2.0.

Everything else — HTTP transports, databases, identity providers, protocol adapters, deployment packaging — is an application concern that belongs outside this library.

---

## What is intentionally out of scope?

The following concerns are explicitly out of scope for this repository and must never enter it:

- HTTP servers, API frameworks, WebSocket or gRPC handlers
- Database clients, ORMs, storage backends
- Identity provider clients (Keycloak, LDAP, OIDC token validation)
- Protocol adapter implementations (BACnet, Modbus, MQTT, OPC-UA)
- Deployment infrastructure (Docker, Kubernetes, cloud SDKs)
- Operator or administrator UI
- Policy authoring tools or configuration languages
- Certificate management or credential lifecycle

These constraints are constitutional (see `docs/kernel-constitution.md`) and are enforced by static import boundary tests in `tests/test_import_boundaries.py`. A change that introduces any of the above is categorically out of scope regardless of how the motivation is framed.

## What public contracts are stabilized?

The following contracts are stabilized for v0.1. Changes to them require architecture review in basis-architecture and, for breaking changes, an ADR and defined migration path. See `docs/breaking-change-discipline.md`.

### Stable public API (`docs/public-api.md`)

Six packages with explicit `__all__`:

| Package | Key symbols |
|---|---|
| `basis_core.domain` | `Subject`, `SubjectType`, `Resource`, `ResourceType`, `IdentityContext`, `build_resource_id`, `parse_resource_id`, `subject_from_jwt`, `action` module |
| `basis_core.decisions` | `DecisionRequest`, `DecisionResponse`, `DecisionOutcome`, `FailureReason` |
| `basis_core.policy` | `PolicyEngine`, `PolicyRule`, `Decision`, `PolicyOutcome`, `RolePolicyRule`, `ResourceTypePolicyRule`, `ActionPolicyRule` |
| `basis_core.audit` | `AuditEvent`, `AuditEventType`, `AuditOutcome`, `AUDIT_SCHEMA_VERSION`, `AuditWriter`, `NullAuditWriter`, `LogAuditWriter`, `DecisionTrace`, `RuleEvaluation` |
| `basis_core.enforcement` | `EnforcementPoint` |
| `basis_core.adapters` | `AdapterBase`, `NormalizedEvent` |

### JSON schemas (`schemas/`)

Four schemas with `additionalProperties: false` and explicit required-field sets:
`decision-request.schema.json`, `decision-response.schema.json`, `audit-event.schema.json`, `policy.schema.json`.

### Evaluation semantics (`docs/evaluation-semantics.md`)

DENY short-circuits; ALLOW does not. First ALLOW wins. NOT_APPLICABLE resolves to DENY at the enforcement boundary. Exception in a rule → fail-closed DENY with `is_error=True`. Empty policy list → DENY. These semantics are covered by 49 contract tests in `test_evaluation_semantics.py`.

### Extension contracts (`docs/extension-contracts.md`)

`PolicyRule`, `AuditWriter`, and `AdapterBase` behavioral contracts, including determinism requirements, forbidden side effects, failure behavior, and isolation constraints. Covered by 68 contract tests in `test_extension_contracts.py`.

### Enforcement boundary contracts (`docs/enforcement-boundary.md`, `docs/failure-modes.md`)

`EnforcementPoint.evaluate()` never raises. All failure paths return DENY. Raw exception text never reaches the caller. `FailureReason` codes are stable serialized strings.

## What compatibility guarantees exist?

**Additive changes** (new optional fields, new enum values, new public symbols, new rule types) may be made without a breaking-change process, provided they satisfy the additive criteria in `docs/breaking-change-discipline.md`.

**Breaking changes** (field removal or rename, required field addition, enum value removal, semantic redefinition, `__all__` removal) require: architecture review in basis-architecture, an ADR, a defined migration path, and deliberate fixture updates. They may not be merged until governance is complete.

**Compatibility test harness** (`docs/compatibility-testing.md`):
- `test_contract_snapshots.py` — serialized model shapes vs. stored fixtures (fail = breaking change detected)
- `test_backward_compatibility.py` — stored fixtures round-trip through current code (fail = existing records broken)
- `test_schema_versioning.py` — JSON schema structural requirements, required-field and enum snapshots
- `test_public_api.py` — `__all__` inventory vs. documented surface (fail = accidental export change)

These tests are the canonical signal that a contract surface has changed. Test failures must not be silenced before governance steps are complete.

## What known open questions remain?

The following questions are intentionally deferred and acceptable for v0.1. Each is tracked with an `OPEN:` marker in the relevant doc.

| Question | Tracked as | Status |
|---|---|---|
| `action` module export pattern (module vs. individual constants) | `OPEN: action-module-export-pattern` | Deferred. Current approach (module re-export) is consistent and avoids namespace pollution. |
| Top-level `basis_core` convenience namespace | `OPEN: top-level-namespace` | Deferred. A flat top-level namespace is ergonomic but commits to a harder-to-evolve shape. |
| `subject_from_jwt` placement in `domain` vs. future `auth` helper | `OPEN: subject-from-jwt-placement` | Deferred. Function works correctly where it is. Placement is an organizational concern, not a behavioral one. |
| `AuditEvent.subject_type` model/schema alignment (open-string model vs. enum schema) | `OPEN: audit-subject-type-alignment` | Deferred. Current state is intentional: open-string model for forward compatibility, enum schema for structural clarity. Low operational risk. |
| Schema `$id` namespace (placeholder vs. resolvable URI) | `OPEN: schema-id-namespace` | Deferred. Must not change until basis-architecture resolves the canonical schema registry question. |
| Schema-level version annotation (`x-schema-version` on schema files) | `OPEN: schema-file-version-annotation` | Deferred. Not blocking for v0.1; becomes important once consumers pin to specific schema revisions. |
| Semantic versioning operationalization across the ecosystem | `OPEN: schema-semver-operationalization` | Deferred. Pre-release; no existing consumer pinning. |

None of these require resolution before v0.1. All are documented with explicit disposition and tracking tags.

## What should NOT be added before v0.1?

Adding the following before v0.1 would delay the release without benefit and risk compromising the kernel's boundary:

- An HTTP server, REST API, or WebSocket handler
- A database-backed `AuditWriter` implementation
- An OIDC/JWT validation integration in `basis_core.domain`
- A BACnet, Modbus, MQTT, or OPC-UA adapter implementation
- A policy configuration file parser or DSL
- A policy administration API
- Async variants of `EnforcementPoint.evaluate()` or `PolicyEngine.evaluate()`
- Cloud deployment tooling or container configuration

These belong in future ecosystem components (`basis-gateway`, `basis-adapters`, `basis-deploy`) that depend on basis-core, not in the kernel itself.

## Remaining work before v0.1 release

The following items should be completed before cutting the v0.1 tag:

### Required

- [ ] **Remove `basis_core.api` stubs** — The `src/basis_core/api/` directory is a deprecated namespace from an internal refactor. The files emit `DeprecationWarning` on import and are scheduled for removal. They cannot be deleted in the current environment due to filesystem constraints; delete them before the v0.1 tag is cut. Remove the corresponding `TestDeprecatedApiPackage` tests and the deprecated surface section from `docs/public-api.md`.

- [ ] **Add a `CHANGELOG.md`** — There is no changelog. v0.1 is a good milestone to establish the baseline entry. A minimal entry stating what v0.1 includes and that it is the initial stable release is sufficient.

- [ ] **Verify PyPI package installability** — Run `pip install basis-core==0.1.0` from a clean environment after publishing. Confirm `import basis_core; basis_core.__version__` works as expected.

### Recommended (but not blocking)

- [ ] **Resolve `OPEN: subject-from-jwt-placement`** — This is the one open question with a possible v0.1 impact. `subject_from_jwt` assumes a Keycloak/OIDC JWT structure and lives in `domain/`. If the expectation is that JWT structure varies across deployments, relocating it to a future `basis_core.auth` helper before v0.1 avoids a later breaking change. Defer only if the current placement is intentional for the initial release audience.

- [x] **`py.typed` marker added** — An empty `py.typed` file has been added to `src/basis_core/` declaring the package as typed per PEP 561. Downstream users and type checkers can use basis-core's type annotations directly without stubs.

## What work is intentionally deferred until after v0.1?

The following work belongs in post-v0.1 releases and should not be pulled forward:

- **`basis_core.api` removal** — Already deprecated with warnings; delete the directory in the next release cycle once the deprecation period is established.
- **Top-level convenience namespace** — `from basis_core import Subject, EnforcementPoint` ergonomics (`OPEN: top-level-namespace`). Requires a stable shape commitment that is better made after v0.1 adoption patterns are understood.
- **Schema `$id` migration** — Must follow canonical schema registry resolution in basis-architecture (`OPEN: schema-id-namespace`).
- **Async evaluation support** — No current blocker, but async variants would introduce interface questions (does `PolicyRule.evaluate()` become `async def`?). Requires an ADR.
- **Policy configuration file format** — A YAML/JSON policy loader would be useful but belongs in basis-gateway, not the kernel.
- **Additional built-in policy rule types** — `TimeBoundPolicyRule`, `ContextualPolicyRule`, etc. can be added additively post-v0.1 without a breaking change.
- **Schema-level version annotation** — (`OPEN: schema-file-version-annotation`) Becomes important as external consumers begin pinning to specific schema revisions.

## Is basis-core suitable for early adopters?

**Yes, with the caveats noted above.**

basis-core has a stable, tested, and documented public API. The evaluation semantics, extension contracts, and enforcement boundary behavior are contract-tested with 625 passing tests. The kernel boundary is enforced both by architectural documentation and by static import tests that fail on violation. A breaking-change governance process is in place.

The library is suitable for early adopters who:
- Are building a gateway, adapter host, or authorization-adjacent service
- Understand that v0.1 is a stabilization milestone, not a production deployment endorsement
- Can tolerate the deprecation of `basis_core.api` (now warns; will be removed)
- Are prepared to follow the breaking-change process if they propose changes to contract surfaces

**Known limitations at v0.1:**
- The `basis_core.api` namespace still exists and must be manually deleted before the tag is cut
- No `CHANGELOG.md` exists yet
- `subject_from_jwt` has an open placement question that could result in a breaking change if resolved post-v0.1

None of these limitations affect correctness or the core authorization contract. They are packaging and lifecycle concerns.
